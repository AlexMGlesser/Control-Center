Current App Runtime Model
- Control Center is now a desktop app runtime using Electron.
- Internal backend is Express and runs locally.
- UI is dark-mode themed and rendered in the Electron window.
- Mobile mode and desktop mode state switching is implemented as a backend state machine.
- Agent runtime uses deterministic intent routing first, with local model fallback for non-deterministic prompts.

Implemented Features (control-app)
1. Desktop shell and navigation
- Sidebar with system tabs and app tabs.
- App tabs are populated from backend app registry.
- Smooth tab traversal and status cards.

2. App registry (WIP modules)
- project-app
- work-app
- calendar-app
- drawing-app
- news-app
- music-app
- server-manager-app
- movie-app

3. Event bus
- In-memory event bus with bounded history.
- API endpoints for listing, posting, and SSE streaming events.
- Dedicated Event Bus tab with fixed-height scroll panel.

4. Mode management
- Backend mode state machine with transition history.
- Header controls to switch Desktop/Mobile mode.

5. Settings tab (desktop-focused)
- Audio device selection.
- Voice selection.
- Voice speed selection.
- Save settings endpoint.
- Test Voice button endpoint (currently local stub behavior + event emission).

6. Chatbot tab
- Text chat available in Desktop Mode.
- Mobile mode shows desktop-required notice.
- Local in-memory chat service and API implemented as integration-ready stub.
- User messages produce local stub agent replies.

7. Startup script
- PowerShell startup script exists in control-app:
  - start-control-center.ps1

8. Calendar app (implemented)
- Standalone Calendar app is served from the root workspace `calendar-app` folder.
- Calendar app opens in its own Electron window when explicitly requested.
- Month grid + selected-day agenda UI is functional in dark mode.
- Backend service persists events to `control-app/server/data/calendar-events.json`.
- Calendar tool operations implemented:
  - read today/tomorrow/week/specific date
  - read rest of day
  - add event
  - remove event

9. Voice pipeline (in progress)
- Whisper + Piper service modules were added under `control-app/server/services`.
- Local voice websocket route is active and clients can connect.
- Voice setup assets/scripts exist under `control-app/voice`.
- Voice stack is functional for connectivity, with tuning/stability work ongoing.

Known Limits (by design)
- LLM/tool-call responses can still vary by phrasing when prompts are far from supported deterministic patterns.
- Voice stack binaries/assets are large and should remain intentionally scoped during commits.
- Some app modules are still intentionally WIP shell states.

Recommended Next Priorities
1. Add recurring events and event editing support to the calendar service and UI.
2. Add deterministic parsing for more natural calendar add/remove phrases and explicit confirmations.
3. Stabilize voice pipeline end-to-end (record, transcribe, route command, speak response).
4. Add persistent settings and chat storage (file/db) instead of memory-only runtime state.
5. Add app-specific event filtering in Event Bus tab.

Conventions Used
- Local-first desktop execution.
- App isolation via module boundaries and app registry.
- Event-driven status updates via event bus.
- Dark-mode style tokens and consistent UI language.
